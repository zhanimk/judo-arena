## 1) Что уже есть в проекте (по коду)

Ниже — быстрый матчинг ТЗ к текущим backend-модулям.

### Реализовано (базово)
- Auth (регистрация/логин/me): `auth.routes.js`
- Пользователи/профили/статусы: `users.routes.js`
- Клубы (создание, заявки в клуб, approve/reject): `clubs.routes.js`
- Турниры (CRUD + статус/публикация): `tournaments.routes.js`
- Заявки (создание, submit, review, approve/reject): `applications.routes.js`
- Сетки (генерация, просмотр сетки и матчей): `brackets.routes.js`
- Матчи (start/score/penalties/finish/reopen): `matches.routes.js`
- Судейские операции (очередь татами, назначение, reorder): `judges.routes.js`
- Live-управление турниром (start/pause/resume/complete): `tournament-live.routes.js`
- Админ-корректировки матча (override/replay/rollback): `admin.routes.js`
- Уведомления: `notifications.routes.js`

## 2) Gap-анализ относительно ТЗ

### A. Критично для MVP (доделать в первую очередь)
1. **Полный e2e для ключевого бизнес-потока**
   - регистрация -> клуб -> заявка -> approve -> генерация сетки -> проведение матчей -> завершение турнира.
2. **Стабильная документация API**
   - единый реестр endpoint + статусы/ошибки + роль-доступ.
3. **Ограничения доступа и ownership-правила**
   - перепроверка, что спортсмен/тренер/судья видят только разрешённые сущности.
4. **Аудит административных вмешательств**
   - верификация, что все override/replay/rollback логируются полно и консистентно.

### B. Функциональные зоны “частично/нужно подтвердить”
1. **Судейский интерфейс таймера**
   - в backend есть управление матчем, но нужен явный контракт таймера (state/event API + socket события).
2. **Ручная корректировка сетки до старта поединков**
   - в ТЗ заявлена операция переноса участников между поединками.
3. **Модуль статистики/архива**
   - нужен набор отчётных endpoint (история спортсмена, статистика клуба, архив турниров).
4. **Категории как отдельная сущность**
   - в ТЗ есть `Category`; в коде требуется формально подтвердить модель/жизненный цикл категории.

### C. Нефункциональные требования
1. **CI quality gates**: lint + test + (опционально) coverage.
2. **Health/readiness endpoint** для деплоя.
3. **Консистентный error format** для всех API.
4. **Минимальный security hardening** (rate limiting, базовые anti-abuse лимиты).

## 3) План реализации (по спринтам)

## Спринт 1 (1–2 дня)
- Зафиксировать “матрицу ТЗ -> endpoint/service/model” в документации.
- Довести API docs до состояния “можно сдать/демонстрировать”.
- Добавить недостающие интеграционные тесты критического потока.

## Спринт 2 (2–3 дня)
- Закрыть ownership/role edge-cases.
- Доработать админ-корректировки + валидацию последствий по сетке.

## Спринт 3 (1–2 дня)
- Нефункционал: health/readiness, security, логи, polishing.
- Финальная демо-подготовка: сценарий защиты, набор скринов, стабильный seed demo-данных.
